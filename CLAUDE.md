## Что это за проект

`~/croissant-model/` — интерактивный финдашборд для идеи импорта замороженных круассанов из ЕС в РК. Один файл `index.html` (~1370 строк, HTML+CSS+JS+Chart.js через CDN), без сборки.

Live: https://daurenbol.github.io/croissant-import/  
Repo: https://github.com/DaurenBol/croissant-import (`main` → Pages)

## С чего начать в новой сессии

1. Прочитай **docs/SESSION_STATE.md** — там последнее состояние работы.
2. Открой **docs/PLAN.md** — что сделано / в работе / в backlog.
3. Открой **docs/MODEL.md** — чтобы быстро вспомнить параметры `A_DEFAULT`, семантику полей в `cm()`, и почему какие-то значения именно такие.
4. Перед любыми правками — открой `index.html` в Chrome, чтобы видеть что получится:
   ```sh
   open -a "Google Chrome" ~/croissant-model/index.html
   ```

## Правила работы

- **Один файл**: всё живёт в `index.html`. НЕ создавай `script.js`, `styles.css` и т.п. — это осознанное решение (простота, авто-деплой, клиент может скинуть файл куда угодно).
- **Чистый HTML/JS, без сборки**: никакого Vite/Webpack/npm. Chart.js — через CDN.
- **Кеш**: уже стоит `<meta http-equiv="Cache-Control" content="no-cache">` в head, поэтому cache-bust через `?v=` не нужен.
- **Сохранение на клиенте**: HR-штат и дорожная карта хранятся в `localStorage` под ключами `croissant_hr_v1` и `croissant_roadmap_v1`. Если меняешь структуру HR/RM_DEFAULT — бампай суффикс ключа (`v1` → `v2`), иначе старые записи в браузере у клиента затрут новые defaults.
- **Палитра**: имена CSS-переменных (`--navy`, `--teal`, `--gold`, `--red`) **оставлены от оригинала Sulvara** — меняются только их значения в `:root`. Это сделано специально, чтобы не трогать 1000+ мест использования в CSS и JS. При смене палитры — меняй только 5-7 строк в блоке `:root`. Подробности — в `docs/DESIGN.md`.
- **Паттерн save-button**: `S` (staged) = значения в слайдерах, `A` (committed) = значения для расчёта. Пока не нажата кнопка «Сохранить допущения» — модель использует старые значения. Это паттерн из оригинала, не трогай.

## Что НЕ делать

- Не копируй цифры обратно в `A_DEFAULT` если пользователь просто «поиграл» слайдерами — они должны править код руками если хотят закрепить.
- Не переписывай `cm()` целиком из-за мелких правок — ядро расчёта очень плотное и хрупкое. Лучше добавь новое поле в `A_DEFAULT` и используй его точечно.
- Не убирай BUG-FIX комментарии из JS (`// BUG2 FIX`, `// BUG12 FIX`, …) — это следы дебага из оригинала Sulvara, они помечают места где были off-by-one / NaN / double-counting, и если их рефакторить без понимания — баги вернутся.
- Не делай из этого Next.js/React проект. Клиент хочет один файл.
