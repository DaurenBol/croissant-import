# MODEL.md — финансовая модель под капотом

Ключ к проекту. Без этого документа разобраться в `cm()` и `A_DEFAULT` трудно, потому что имена полей **унаследованы от Sulvara Prospera** (завод H₂SO₄) — некоторые имена не соответствуют смыслу.

## v2.0 — multi-SKU модель (с 2026-04-25)

Бизнес возит **ассортимент** разных продуктов (круассаны / слойки / пирожные…), а не моно-SKU. Каждый продукт имеет свою цену/закупку/потери, разные продукты делят объём фуры по долям. Главная метрика для решения «какой микс выгоднее» — **маржа на кг** (фура ограничена по весу).

### Архитектура расчёта

Три уровня функций:

1. **`cm(a)`** — легаси, оригинальное single-SKU ядро. Не трогаем (там BUG-FIX'ы из Sulvara). Может вызываться отдельно для совместимости.
2. **`cmSku(a, sku)`** — расчёт **одного SKU**: vol_kg, units, выручка, переменные затраты (закупка + фрахт + пошлина + лог.РК + хранение). **Без общих расходов** (ФОТ / админ / CAPEX / амортизация).
3. **`cmMulti(a)`** — главная функция. Для каждого SKU вызывает `cmSku()`, **суммирует** результаты, **один раз сверху** добавляет общие постоянные (`fT`, `am`, `fF`) и CAPEX. Возвращает `{ perSku, vol, rev, vT, fT, am, oc, eb, et, kp, np, ph2y, psy, pey, ky, capexM, amort, fot, lossEff }`.

`cmMulti()` возвращает поля совместимые с `cm()` — поэтому все старые render-функции продолжают работать без изменений, читая total. Дополнительно появляется `perSku[]` для детализации.

### A_DEFAULT — новые поля (v2.0)

| Поле | Дефолт | Смысл |
|---|---|---|
| `trucks_per_year` | 10 | Сколько фур (рефрижераторов) приходит в год — глобальный объём |
| `truck_kg` | 20000 | Полезная загрузка одной фуры, кг |
| `skus[]` | 3 SKU | Массив продуктов |

Объём каждого SKU в кг/год: `vol_kg = trucks_per_year × truck_kg × sku.share × A.load[i]` (load — общий рост 5 лет).

### A_DEFAULT.skus[] — структура SKU

| Поле | Пример | Смысл |
|---|---|---|
| `id` | `sku_croissant` | Уникальный ID (для React-keys и localStorage) |
| `name` | `'Круассан классик'` | Название |
| `price_kzt_unit` | 280 | Отпускная цена, тг/шт |
| `purchase_eur_kg` | 4.0 | Закупка у поставщика, €/кг (стандарт для пищёвки) |
| `weight_g` | 80 | Вес 1 шт, грамм |
| `loss_pct` | 0.035 | Потери по сроку годности, доля |
| `share` | 0.6 | Доля SKU в общем весе фуры (сумма всех share = 1.0) |

**Производные** (вычисляются на лету): `price_kzt_kg = price_kzt_unit / weight_g × 1000`, `units_year = vol_kg / weight_g × 1000`.

### Метрики на трёх уровнях

| Уровень | Метрики | Что говорит |
|---|---|---|
| **Per SKU — контрибуция** | `cm` (тг), `cmPct` (%), **`cmPerKg`** (тг/кг) | Выгодно ли держать SKU в миксе. **`cmPerKg` — ключ к оптимизации**: фура весовая, поэтому SKU с большей маржой за кг должен иметь большую долю |
| **Per SKU — выручка** | `rev`, `units` (шт/год), `vS/vE/vW` | Сколько штук продаём, какая выручка, структура переменных затрат |
| **Бизнес целиком** | NPV, IRR, payback, PI, BEP, EBITDA, NP | Решение «инвестировать или нет». **Только на этом уровне** — CAPEX и ФОТ общие, делить по SKU = искусственно |

### Что НЕ меняли

- `cm()` legacy — нетронут, все BUG-FIX'ы (BUG2, BUG5, BUG12) сохранены
- HR / Risks / Roadmap / Invest — без изменений (CAPEX считается на весь бизнес)
- DDS — пересчитывается через `cmMulti()` total, формат тот же

## Формулы — функция `cm(a)`

Принимает `A` (ассумпции) или модифицированный объект (для сценариев/чувствительности). Возвращает объект с массивами по 5 годам.

### Выручка

```js
sold[i] = vol[i] × (1 − rvoda)              // проданный кг (за минусом списаний)
ph2y[i] = ph2 × (1 + ih2)^i                  // цена продажи с эскалацией
rev[i]  = sold[i] × ph2y[i]                  // выручка в тг
```

### Переменные затраты

```js
// Цены в тенге с эскалацией
ky[i]   = kurs × (1 + ikurs)^i               // курс EUR по годам
psy[i]  = psera_usd × ky[i] × (1 + isera)^i  // закупка тг/кг
pey[i]  = pel × ky[i] × (1 + iel)^i          // фрахт тг/кг
fobPerKg[i] = psy[i] + pey[i]                // CIF-эквивалент для расчёта пошлины

// Компоненты
vS[i] = vol[i] × psy[i]                                            // закупка товара
vE[i] = vol[i] × (pey[i] + fobPerKg[i] × rsera + fobPerKg[i]×0.015) // фрахт + пошлина + 1.5% брокер
vW[i] = vol[i] × (pvoda × (1+inf)^i + pteplo × (vteplo/30) × (1+inf)^i)  // лог.РК + хранение

vT[i] = vS[i] + vE[i] + vW[i]
```

**Важно**: затраты считаются от **полного объёма** `vol[i]`, а не от `sold[i]`. Списанный товар тоже был куплен и растаможен. Это основной источник расхождений с наивными расчётами в Excel.

### Постоянные затраты

```js
fF[i] = fot × (1 + nrabot) × (1 + inf)^i     // ФОТ с налогами + эскалация
capexM = capex_usd × kurs + capex_kzt         // всего CAPEX в млн тг
amort = capexM / amort_spi                    // годовая амортизация (млн)

fT[i] = fF[i] + (to + strah + zeml) × MLN + adm × MLN × (1+inf)^i + okhrana × MLN × (1+inf)^i
```

### EBITDA → Чистая прибыль

```js
oc[i] = vT[i] + fT[i]                         // OPEX total
eb[i] = rev[i] − oc[i]                        // EBITDA
et[i] = eb[i] − am[i]                         // EBIT
kp[i] = max(et[i] × kpn, 0)                   // КПН (не может быть отрицательным)
np[i] = et[i] − kp[i]                         // Чистая прибыль
```

### FCF и Инвест-метрики

- **FCF₀** = −CAPEX (в t=0, год «2025»)
- **FCF_t** (t=1..5) = NP_t + Аморт_t − ΔNWC_t (NWC вшит как массив `[0, -50, -80, -20, -20]` в `autoFlows()`)
- **NPV** = Σ FCF_t / (1+WACC)^t
- **IRR** — ньютоновским методом с fallback-стартами (borrowed from оригинала, уже отлаженный)
- **PP (payback)** — простой (не дисконтированный). BUG2-FIX в коде — не трогать.
- **PI** = 1 + NPV/CAPEX
- **BEP** = FC/(P − VC/кг) — точка безубыточности в кг/год

## Что не учтено (упрощения)

- **НДС ввозной** числится в кэш-потоке, но не влияет на P&L (через 1-3 мес возмещается). В ДДС `autoFlows.receipts` стоит `rev × 1.12` (вытягивающий НДС с продаж) — грубое приближение.
- **Возврат NWC** в конце горизонта — не учитывается (можно добавить +NWC в FCF_5, тогда IRR станет оптимистичнее).
- **Равномерная загрузка** — продажи в модели равномерно по году. Для реальности нужно ramp 1-го года (есть в `renderPL`) и сезонность (нет, можно добавить).
- **Каналы сбыта** — показаны в `A_DEFAULT` как `ch_retail/horeca/own` (но эти поля не задействованы в `cm()`, пока декоративны). Если клиент захочет видеть P&L по каналам — это апгрейд.
- **Собственная розница** — не выделена как отдельный бизнес. Если появится, нужно отдельный CAPEX + OPEX + отдельный P&L.

## Баги-фиксы, перенесённые из оригинала

В коде есть комментарии `// BUG2 FIX`, `// BUG5 FIX`, `// BUG9 FIX`, `// BUG12 FIX` — это исправления из Sulvara Prospera. **НЕ УДАЛЯТЬ** — они помечают места где были off-by-one, NaN в IRR, double-count CAPEX и т.п. Ядро расчёта хрупкое.
