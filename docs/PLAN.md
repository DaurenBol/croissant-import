# PLAN.md — план развития проекта Croissant Import

## Цель

Готовый интерактивный дашборд-презентация для клиента, который хочет посмотреть на бизнес-идею «импорт круассанов в Казахстан» в том же формате, что и просчёт завода H₂SO₄ (Sulvara Prospera).

### 🚀 v2.0 — multi-SKU модель (в работе с 2026-04-25)

Принципиальный пересмотр единицы учёта: **с «кг круассанов» на «штуки разных SKU»**. Импортёр везёт не моно-продукт, а ассортимент (круассаны, слойки, пирожные…), каждый со своей ценой/себестоимостью/долей в фуре. Бизнес-смысл: предприниматель должен видеть, **какой микс ассортимента максимизирует маржу** при заданной загрузке фур.

**Ключевые изменения:**

- [ ] `A_DEFAULT.skus[]` — массив SKU. Каждый: `{ name, price_kzt_unit, purchase_eur_kg, weight_g, loss_pct, share }`. Дефолт — 3 SKU (круассан классик / шоколадный / слойка)
- [ ] `A_DEFAULT.trucks_per_year` + `truck_kg` — глобальный объём импорта в фурах. Объём каждого SKU = trucks × truck_kg × share / 1
- [ ] `cmSku(A, sku)` — расчёт переменных затрат и выручки одного SKU. Без общих расходов
- [ ] `cmMulti(A)` — обёртка: для каждого SKU зовёт `cmSku()`, агрегирует + один раз сверху добавляет ФОТ/админ/охрана/CAPEX/амортизацию. Возвращает `{ perSku, ...total }`. Старые render'ы переключаются на неё
- [ ] **Маржа на кг (тг/кг)** — главная метрика для оптимизации микса. Фура ограничена по весу, поэтому SKU с маржой 200 тг/шт × 60 г = 3333 тг/кг лучше, чем 250 тг/шт × 100 г
- [ ] UI блок «Ассортимент» в Допущениях: таблица SKU с инлайн-полями, ползунки долей с автонормализацией к 100%, кнопки `+SKU` / `×`. Лимит 8
- [ ] Live-таблица «Маржа по SKU» под настройками — обновляется на лету, лучший SKU зелёным, худший красным
- [ ] Дашборд — переключатель «Общий бизнес / По продукту»: stacked bar по SKU; инвест-метрики (NPV/IRR/payback/BEP) только на общем уровне
- [ ] 3 пресета микса: «Только круассаны» / «Сбалансированный» / «Премиум-микс»
- [ ] `localStorage croissant_skus_v1` + миграция: если есть старые ключи, создаём 1 SKU «Круассан» из них
- [ ] Сценарии и чувствительность — применяют коэффициенты ко всем SKU одновременно

**Что НЕ меняем:**

- Старая `cm()` — остаётся как легаси для совместимости (вдруг где-то всплывёт). Все BUG-FIX'ы переносятся в `cmMulti()` без модификации
- Глобальные расходы (CAPEX, ФОТ, админ, страхование, охрана) — общие на бизнес. НЕ размножаем по SKU. Это сознательное решение: инвестиция одна, делить искусственно бессмысленно
- HR / Дорожная карта / Риски / Инвестиционный обзор — без изменений

**Бэкап перед рефакторингом:**

- Файл: `backups/index_v1_singleSKU_kg_2026-04-25.html`
- Git тег: `v1-single-sku-kg` (локально, не запушен)

### 🔄 v1.1 — калибровка цифр (отложено до v2.0)

Текущие defaults в `A_DEFAULT` — разумные, но **не валидированные**. Перед показом инвестору нужно:

- [ ] Проверить **актуальную ставку пошлины** ТН ВЭД 1905 90 — через «Консультант» или ТС РК (сейчас стоит 12.5%)
- [ ] Уточнить **реальные цены закупки** у 2-3 французских/польских поставщиков (FCA/CIP)
- [ ] Уточнить **стоимость фрахта** через рефрижератор ЕС → Алматы (€/кг, срок, минимальный объём)
- [ ] Проверить **ставку аренды холодильного склада** в Алматы/Астане (м²/мес, −18°C)
- [ ] Уточнить **отпускные цены** — опросить 2-3 сети (Magnum/Small/Galmart) на предмет того, сколько они готовы платить за кг замороженного круассана
- [ ] Реалистично прикинуть **объём продаж** через разные каналы (сети / HoReCa / собственная розница)

**Как корректировать**: правь значения в `A_DEFAULT` (блок в `index.html` ~строка 355), делай коммит, пушь — через минуту обновится live.

### 🚫 Намеренно НЕ делаем

- **Next.js/React-версия** — один файл HTML проще для клиента (перекинул по Telegram, открыл, всё работает).
- **Backend / база данных** — все «сохранения» живут в localStorage. Если клиент захочет мульти-юзер + синхронизацию — это отдельный продукт (тут пригодится существующий `~/import-control-v3/` с NextAuth + Prisma).
- **Отдельные JS/CSS файлы** — осознанный выбор держать всё в одном `index.html`. При декомпозиции теряется плюс «один файл = вся модель».

## Взаимодействие с клиентом

Workflow при правках после демо:

1. Клиент говорит: «NPV красный, сделайте оптимистичнее» / «поднимите цену продажи до 3500»
2. Открываем `index.html`, правим `A_DEFAULT` (или конкретную формулу в `cm()`)
3. Открываем в Chrome, проверяем
4. `git add -p && git commit && git push`
5. Клиент обновляет страницу через 60 сек — видит новую модель

## Риски проекта (для самого себя, не для клиента)

- **Кэш Pages**: был случай что CDN GitHub Pages кеширует старый HTML. Если после пуша ничего не поменялось — hard reload (Cmd+Shift+R).
- **localStorage-конфликты**: если клиент уже открывал сайт и редактировал HR/дорожную карту — при смене `HR_DEFAULT`/`RM_DEFAULT` его версия перевесит defaults. Лечится бампом ключа (`croissant_hr_v1` → `v2`).
- **Замена Chart.js CDN**: если jsdelivr упадёт — графики не нарисуются. Запасные CDN: unpkg, cdnjs. Менять в `<head>`.
